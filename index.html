Testing: Fitness App
Test Scope Test Plan Phase One Phase Two Phase Three Validation Testing

Test Scope

The service we will be testing is a fitness app that follows a freemium model. This means the application is free to use, but has premium features that are locked behind a paywall. This application has three core software features:

1. A list of videos that can be filtered by labels (e.g., length, body part, and whether the video is available to all users or premium-only).

2. The ability for users to upgrade and downgrade their accounts.

3. Performance charts that have limited data visibility, depending on whether the user has a free or premium account.

This document will cover testing strategies for functional, performance, and internal design characteristics of the high-level design presented. More specifically, this document will look at:

• The integration of the product's video filtering function and video filter user interface

• The functionality of the product's account management features

• The functionality differences between 'free' and 'premium' accounts performance tracking

• Validation testing to ensure functional and non-functional requirements are met

Test Plan

This testing strategy will follow both a top-down and bottom-up integration testing approach where the testing will be conducted in 3 separate phases. The decision to use these approaches was made since the tests will be based on a high-level design. The high-level design of the system highlights the system's overall structure and main modules, and user workflows. A top-down approach allows for testing of the main modules and user-visible features early on. At the same time, the bottom-up testing approach ensures that the main modules are thoroughly validated. By combining both of these testing approaches, we are able to analyze both the high and low-level aspects of the system for optimal coverage.

The system has a modular architecture that has three main modules: Video List Filter Module, Account Upgrade/Downgrade Management Module, and Performance Chart View Module. The decision to break the integration testing into three phases was made since it aligns with the system's architecture.

Phase 1 will consist of testing the Video List Filter Module. It will test the video list filter functionality and integration with a user interface. This phase is necessary as it isolates content discovery and filtering functionality, which is foundational to the user experience. Phase 2 will test the Account Upgrade/Downgrade Management Module, which is the account tier management functionality. This phase is necessary as the account upgrade/downgrade process is central to the user experience, as it enables premium-level features. The final phase will test the Performance Chart View Module, which is the fitness metrics visualization. This module is dependent on the proper functioning of the account management system, which is why it is the last phase.

Phase 1: Video Filtering and User Interface Integration Testing

In phase one, our team is focusing on validating the filtering functionality within the fitness app, ensuring that users can seamlessly apply, remove, and clear filters based on different criteria such as video labels, genres, and account types. These tests are designed to confirm that the app accurately filters content based on user preferences and subscription tier, ensuring that free-tier and Pro-tier users receive the appropriate content. Additionally, we are examining the app's navigation flow, ensuring that users can easily log in, navigate through the app, and interact with the video filtering system without issues. The table below outlines the specific test cases designed to validate the app's filtering capabilities, UI responsiveness, and navigation functionalities, ensuring a smooth user experience.

Test Case ID Test Case Objective Test Case Description Expected Results
TC001 Add a filter by label Add a single label filter using the FilterByLabelCommand. The specified label filter is added to the FilterManager. Observers are notified.
TC002 Remove a specific label filter Remove a specific label from the filter list. The specified label filter is removed. Observers are notified of the update.
TC003 Clear all filters Clear all currently applied filters using the ClearAllFiltersCommand. All filters are cleared. Observers (e.g., UI and API) are notified.
TC004 Video filtering for pro vs free accounts Log in with a free account, navigate to the video list, and ensure only free videos are displayed. Free account users only see videos labeled as free. Pro videos are hidden.
TC005 Video filtering for pro accounts Log in with a pro account, navigate to the video list, and ensure both free and pro videos are displayed. Pro account users see all videos labeled as free and pro.
TC006 Confirm video filtering based on account Apply filters by genre, length, and account type, and validate the resulting video list matches expectations. Video list accurately reflects the selected filters and account type.
TC007 Navigation flow from login to homepage Log in with valid credentials and navigate through the homepage to the video filtering screen. Successful navigation to the video filter screen. UI is responsive and functional.
TC008 Navigation with invalid login credentials Attempt to log in with invalid credentials and verify the user cannot access the video filter screen. User is blocked from proceeding. An appropriate error message is displayed.
TC009 Undo after clearing all filters Clear all filters using ClearAllFiltersCommand and then undo the operation. All previously cleared filters are restored. Observers are notified of the rollback.
TC010 Verify UI updates on filter application Apply multiple filters and confirm the video list updates dynamically in the UI without requiring a page reload. The video list updates in real time to reflect the applied filters.
Table 1: Video Filtering and User Interface Integration

Phase 2: Account Management Functionality Testing

In this testing phase, our team is validating the functionality of account tier transitions within the fitness app, focusing on the user interface elements that manage the visibility of upgrade and downgrade buttons. These tests ensure that users can easily upgrade or downgrade their accounts between free-tier and pro-tier states, and that the UI reflects these changes in real-time. By simulating upgrades, downgrades, and invalid states, we aim to confirm that the app correctly manages account state transitions and displays the appropriate buttons to users. Additionally, we are testing the execution of commands triggered by these buttons, ensuring that the underlying logic is functioning as expected. The following table outlines the specific test cases designed to verify the proper handling of account state changes, button visibility, and the related user interface behavior.

Test Case ID Test Case Objective Test Case Description Expected Results
TC001 Verify upgrade button visibility Ensure the upgrade button is displayed when a free-tier user attempts to access pro content. The upgrade button is visible for the user attempting to access pro content.
TC002 Verify downgrade button visibility Ensure the downgrade button is displayed when a pro-tier user accesses account settings. The downgrade button is visible in account settings for pro-tier users.
TC003 Validate state transition: Free to Pro Simulate an upgrade action for a free-tier user and verify the account transitions to the ProState. The user account transitions to ProState after a successful upgrade.
TC004 Validate state transition: Pro to Free Simulate a downgrade action for a pro-tier user and verify the account transitions to the FreeState. The user account transitions to FreeState after the successful downgrade.
TC005 Validate button visibility logic Verify that only one of the upgrade or downgrade buttons is displayed based on the user's current state. Only the relevant button (upgrade or downgrade) is visible based on the user’s account state.
TC006 Validate Observer Pattern implementation Ensure that UI elements update dynamically when the user's account tier changes (Free to Pro or Pro to Free). UI updates in real-time, showing the correct button visibility upon account state change.
TC007 Validate command execution for UpgradeCommand Execute the upgrade button and verify that the upgrade action is successfully triggered. UpgradeCommand executes successfully, changing the user’s tier to ProState.
TC008 Validate command execution for DowngradeCommand Execute the downgrade button and verify that the downgrade action is successfully triggered. DowngradeCommand executes successfully, changing the user’s tier to FreeState.
TC009 Verify ButtonVisibilityManager functionality Ensure that the ButtonVisibilityManager appropriately toggles the visibility of buttons based on the user's account tier. Buttons are correctly hidden or displayed based on the ButtonVisibilityManager logic.
TC010 Validate error handling for invalid state Simulate a scenario where neither FreeState nor ProState is assigned and verify error handling. The system logs an error and prevents access to features without a valid state assignment.
Table 2: Account Management Functionality Testing

Phase 3: Performance Chart with User Data Testing

For phase three of testing, our team is focusing on the integration of key functionalities in the fitness app to ensure that all features are working seamlessly for both free-tier and Pro-tier users. This testing validates the user experience by ensuring that the app’s performance chart and subscription management features function as expected, providing users with appropriate access based on their subscription tier. Additionally, we are testing the app’s responsiveness to subscription changes, ensuring that users can view and interact with the app according to their subscription status. The following key areas from the app's feature set will be tested: restricted access to chart sections based on subscription tier, visibility of recent activity and detailed metrics, proper functioning of the FullView and LimitedView components, and accurate notification and handling of subscription changes. The table below outlines the specific test cases for these requirements.

Test Case ID Test Case Objective Test Case Description Expected Results
TC001 Verify restricted sections of the chart are hidden for free-tier users. Simulate a free-tier user attempting to access sections of the performance chart reserved for Pro-tier users. Restricted sections are not visible.
TC002 Verify Pro-tier users can access all sections of the chart. Simulate a Pro-tier user accessing the performance chart. All chart sections, including historical data and detailed metrics, are fully visible.
TC003 Verify recent activity display for free users. Render the LimitedView component and check if only the last week’s data is displayed. The component displays only recent activity data from the past week.
TC004 Verify Pro-tier users do not see the LimitedView component. Attempt to render the LimitedView for a Pro-tier user. The LimitedView component is not rendered; instead, the FullView component is displayed.
TC005 Verify historical data and detailed metrics for Pro users. Render the FullView component and ensure it includes historical data, trends, and detailed metrics. The component displays all available chart data for Pro users.
TC006 Verify FullView component is not available for free-tier users. Attempt to render the FullView for a free-tier user. The FullView component is not rendered; the LimitedView component is displayed instead.
TC007 Verify observers are notified of subscription changes. Change the user’s subscription status and ensure all attached observers are notified. Each observer reacts appropriately to the subscription status change.
TC008 Verify observer attachment and detachment functionality based on subscription tier changes. Simulate changes to a user’s subscription tier and confirm that the ChartManager correctly attaches or detaches observers. The user is on the Free tier, only the LimitedView-related observers are active, and FullView-related observers are detached. When the user upgrades to the Pro tier, the FullView-related observers are attached, and the LimitedView-related observers are detached.
Table 3: Performance Chart with User Data Testing
